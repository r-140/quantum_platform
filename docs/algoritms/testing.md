# Тестирование

## Юнит-тесты для `polling.py`

`services/quantum-core/tests/unit/`:
- `fakes.py` — `ScriptedBackend`, тестовый дублёр `QuantumBackend`,
  проигрывающий заданную последовательность статусов и делегирующий
  `fetch_result()` кастомной функции (чтобы симулировать transient-ошибки,
  жёсткие отказы, зависшие job'ы и т.д.);
- `conftest.py` — фикстура `fake_clock`, подменяющая `time.monotonic()` и
  `asyncio.sleep()` внутри `polling.py`, чтобы тесты на backoff/таймаут
  выполнялись за миллисекунды, а не за реальные секунды/десятки секунд;
- `test_circuit_breaker.py` — 5 тестов: открытие по порогу, сброс счётчика
  успехом, half-open переход по времени, повторное открытие после
  неудачной пробной попытки;
- `test_wait_for_result.py` — 8 тестов: немедленный успех, рост интервала
  backoff, retry при transient-ошибке (успешный и исчерпанный), таймаут
  (+ вызов `cancel()`), отмена через `CancellationToken`, "жёсткий" отказ
  (возвращается нормально, не кидает исключение), блокировка при открытом
  circuit breaker (без обращения к backend вообще).

### Как проверялись эти тесты, не имея доступа к pytest

В рабочей среде, где эти тесты были написаны, нет сети для установки
`pytest`/`pytest-asyncio`. Поскольку `polling.py` и `base.py` не зависят
ни от чего, кроме стандартной библиотеки, было возможно:

1. Сначала прогнать все сценарии как обычный asyncio-скрипт (без pytest,
   без фреймворка мокирования) — вручную реализовав тот же `ScriptedBackend`
   и `FakeClock`, что потом легли в `fakes.py`/`conftest.py`.
2. После того как все сценарии прошли, перенести их в pytest-совместимую
   форму (`test_*.py`, фикстуры, `pytest.raises`/`pytest.approx`).
3. **Дополнительно** — прогнать уже сами итоговые файлы тестов (не
   черновик), вручную вызывая тестовые функции и подставляя `fake_clock`
   вместо настоящей pytest-инъекции, с минимальной локальной заглушкой
   `pytest.approx`/`pytest.raises` (заглушка функционально эквивалентна,
   но не является настоящим pytest).

Это даёт высокую уверенность в **логике** тестов и самого `polling.py` —
включая тонкие случаи вроде "жёсткий отказ не должен кидать исключение" и
"открытый breaker не должен даже обращаться к backend", которые легко
перепутать при написании.

### ⚠️ Что осталось не проверено

Сама механика pytest — автоматическое обнаружение фикстур по имени
параметра (`fake_clock`), работа `pytest-asyncio` в режиме `asyncio_mode =
"auto"` (запуск `async def test_...` без дополнительных декораторов),
корректность relative-импорта `from .fakes import ...` при сборе тестов
pytest'ом — **не проверялась**, так как для этого нужен настоящий pytest.
Это стандартный, хорошо документированный паттерн использования, поэтому
риск ниже, чем для логических ошибок, но всё же:

**Прогони `pytest` в своём окружении первым**, прежде чем считать эти
тесты полностью готовыми:

```bash
cd services/quantum-core
pytest tests/unit/ -v
```

Если возникнут проблемы именно с обнаружением фикстур/коллекцией тестов
(а не с логикой самих проверок) — пришли вывод, разберёмся отдельно от
уже подтверждённой логики.
