# Отложенные части архитектуры

Эти куски были в самом первом наброске структуры репозитория (до того, как
начали писать реальный код) и до сих пор не реализованы. Директории-заготовки
под них удалены из дерева проекта — пустая папка без объяснения хуже, чем её
отсутствие; сама идея и обоснование остаются здесь.

## Kafka-телеметрия: `telemetry-ingest` + `stream-analytics`

Из самого первого архитектурного разговора этого проекта: RabbitMQ — для
task queue ("выполни это один раз"), Kafka — для потока телеметрии
("много событий, real-time агрегация, история/replay").

Сейчас `orchestrator/app/tasks/calibration.py` публикует `error_rate` в
**RabbitMQ**-очередь `calibration-results` — явно задокументированная
**временная замена**, а не финальное решение. Когда дойдём до Kafka:

- **`telemetry-ingest`** — продюсер, пишущий события измерений/калибровки
  в Kafka-топик. По факту — `calibration.py` уже делает эту работу
  наполовину; переход означает замену `publish_calibration_result()`
  (сейчас публикует в RabbitMQ) на Kafka producer, без изменения
  `run_calibration()` самого — это уже заложено в текущем коде.
- **`stream-analytics`** — Kafka Streams/Faust, real-time агрегация
  потока телеметрии (rolling error-rate по backend'ам, аномалии,
  алерты при деградации калибровки). Требует, чтобы сначала появился сам
  поток данных (`telemetry-ingest`), поэтому не имеет смысла раньше него.

**Триггер для перехода**: как только `calibration.py` перестанет быть
единственным источником телеметрии (например, если появится второй
backend, или calibration будет гонять чаще одного раза в 5 минут, или
понадобится история/replay событий, а не только текущий снимок) —
RabbitMQ-очередь для этого перестанет быть подходящим инструментом (она
не для этого спроектирована — см. `docs/architecture/orchestration.md`,
"Почему RabbitMQ, а не Kafka").

## `fast-control` — низколатентный control loop (Rust/Go)

Идея из самого первого разговора: тайминг-критичная часть hw/sw
interaction loop (реальное железо может требовать латентность в
микросекундах, пока Python-сервисы работают на уровне более высокоуровневой
оркестрации) — изолированный сервис на Rust/Go, а не размазанный по
Python-коду ради галочки "мы используем Rust".

Не реализован, потому что вся текущая инфраструктура работает с
`AerBackend` (локальный симулятор, без реальных ограничений по
латентности) — `fast-control` имеет смысл проектировать только когда
появится реальный или хотя бы реалистично эмулированный источник
тайминг-давления. `MockHardwareBackend` уже частично играет эту роль
(искусственная латентность/сбои), но пока недостаточно требователен к
реальному времени отклика, чтобы оправдать отдельный Rust-сервис.

## `orchestration/dags/calibration_then_run.py` — gate execution on calibration freshness

Из самого первого наброска: Prefect/Airflow DAG для многошаговых
экспериментов ("калибровка → запуск → анализ").

Сейчас калибровка (`calibration.py`) и запуск эксперимента (`experiments`
очередь) — **независимые, несвязанные потоки**: calibration крутится по
расписанию сама по себе, эксперимент запускается независимо от того,
когда была последняя калибровка и насколько она была "здоровой".

Нереализованная, но реальная идея: **не запускать эксперимент** (особенно
дорогой, вроде VQE), если последняя calibration:
- была слишком давно (backend мог "уйти" с тех пор), или
- показала `error_rate` выше приемлемого порога.

Это потребовало бы: (а) API/orchestrator читал последний
`CalibrationResult` перед постановкой задачи в очередь (или сам
`worker.py` проверял свежесть перед `execute_task`), и (б) хранилище
последнего известного состояния калибровки, доступное обеим сторонам —
естественное место для Postgres/Redis, когда до них дойдём, а не для
отдельного DAG-движка (Prefect/Airflow — оверкилл для одной проверки
"свежести" перед запуском; тот полноценный DAG-движок скорее пригодился
бы для настоящих многошаговых сценариев вроде "откалибруйся → прогони
серию VQE с разными bond length → построй кривую потенциальной энергии").
